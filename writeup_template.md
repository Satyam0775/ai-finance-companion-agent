## 1. Memory: What was stored after Session 1, and what was deliberately not stored?

**Stored:**

| Fact | Why it belongs in memory |
|------|--------------------------|
| User wants to reduce food delivery spending by ~50% | A behavioural commitment — meaningful across future sessions |
| User committed to setting aside ₹30,000 for house fund in November | An explicit decision the user made; the agent needs to recall this |
| Long-term goal: save ₹15 lakh in 2 years for house down payment in Bangalore | The user's stated financial goal — context for every future decision |
| Reminder set for 2025-11-25 to transfer ₹30,000 to house fund | A concrete action the agent took; worth tracking |

**Deliberately not stored:**

| Data | Why it was excluded |
|------|---------------------|
| Checking account balance (₹1,28,000 in Session 1) | Will be different by Session 2 — quoting stale balance could mislead |
| Food delivery total for October (₹12,240) | A point-in-time snapshot; the behavioural pattern matters, not the number |
| Upcoming bill amounts | Could change between sessions; always fetched live |
| Salary amount | Already encoded in the static user profile; doesn't need to live in memory |

**Why the distinction matters:**
Memory that ages badly becomes a liability. If the agent quoted the Session 1 balance in Session 2, it would be giving wrong financial advice. The architecture enforces this by instructing the LLM that balances and totals must always come from fresh tool calls — memory is for enduring facts only.

## 2. Tools vs. LLM: One decision given to the LLM, one kept as code

**Given to the LLM — which tools to call:**
In Session 2, the agent receives a question about buying a MacBook. Nothing in that message explicitly asks about balances or bills. The decision to proactively call `get_account_balance()`, `get_upcoming_bills()`, and `set_reminder()` is judgment — it depends on connecting the new question to the house-fund commitment from memory. That reasoning belongs to the LLM.

**Kept as code — all arithmetic:**
Summing food delivery spend across transactions, halving it for the target, totaling upcoming bills, computing checking-after-bills — all of this lives in `tool_executor.compute_summaries()`. This is deterministic Python. The LLM receives pre-computed numbers and uses them in its prose; it never adds or multiplies anything itself. LLMs can hallucinate arithmetic; Python cannot.

## 3. AI usage: What was AI-generated, and one rejected suggestion

**AI-generated:**
The `RESPONSE_PROMPT` structure — specifically the decision to return a JSON object with `{"response": ..., "memory_updates": [...]}` in a single call — was shaped with AI assistance. This avoids a second "what should I remember?" LLM call and keeps the architecture to two LLM calls per turn (tool decision + response).

**One suggestion that was rejected:**
The AI suggested adding a `"confidence": 0.9` field to each tool decision, so the agent could skip low-confidence tool calls. This was rejected for three reasons:
1. A confidence score generated by an LLM about its own decision is not calibrated signal — it would just add noise.
2. It violates the minimal-code constraint and creates parsing fragility for no measurable benefit.
3. The tool call is binary: either you need the data or you don't. Partial-confidence tool avoidance is the wrong abstraction here.

## 4. One week more: What one thing would be redesigned?

**The memory structure.**

Currently, memory is a flat list of strings — simple and readable, but brittle at scale. With another week, I would structure it as typed categories:

```json
{
  "goals":               ["Save ₹15 lakh in 2 years for house down payment"],
  "commitments":         ["Set aside ₹30,000 for house fund in November 2025"],
  "behavioural_targets": ["Reduce food delivery spend by ~50%"],
  "reminders_set":       [{"date": "2025-11-25", "content": "Transfer ₹30,000 to house fund"}]
}
```

**Why this matters:**
1. The agent could inject only the relevant category into each prompt (goals for big decisions, reminders for follow-ups), reducing token use and increasing precision.
2. It enables contradiction detection — if the user sets a new food delivery target, the old one should be updated, not appended. With a flat list, contradicting facts accumulate silently.
3. It makes the memory file self-documenting and easier to inspect when debugging.
